## 一、模块定位与全景

### 1.1 一句话
**VideosFlow 影片模块 = 一站式"电影/电视剧/综艺/动漫"智能解说制作工作台**，覆盖"从视频到成片"全流程：
- 上传视频 → AI 自动看视频 → 生成匹配画面的解说文案 → 分镜编辑 → 配音 → 导出

### 1.2 VideosFlow 三大模块中的位置

| 模块 | 定位 | 关键能力 |
|------|------|----------|
| **影片** | 类型化工程库 + **成片剪辑 + 智能解说** | 智能粗剪 / 时间线精修 / 字幕花字 / 导出（本次增强） |
| 口播 | 净化 + 花字 | ASR 净化 / 纠正 / 关键词 |
| 创作 | 从需求生成 | 写文案 / 分镜 / 图片 / 视频 / 配音 / 字幕 |

### 1.3 参考软件 vs VideosFlow 影片模块

| 项 | 参考软件（解说猫 v5.2.1） | VideosFlow M2 + M2.5 |
|----|--------------------------|---------------------|
| 视频输入 | ✅ 多选 + 范围裁剪 | ✅ 单选（M2.5 待补批量） |
| 解说风格 | ✅ 15 种（电影/电视剧/综艺/动漫/短剧漫剪/高能合辑/口语化/萌讲/直播/新闻/吐槽/深度解析/慢节奏/悬疑/上帝视角） | 沿用 M3 12 种（M2.5） |
| 场景检测 | ✅ TransNetV2（深度学习 GPU） | ❌ 不做（M6 评估） |
| 视频分析 | ✅ Gemini 2.5 视觉 API | ❌ 不做 |
| **解说生成** | ✅ Gemini 流式 + 六段式 | ✅ **M2.5 ASR + Agnes**（XiaomiMimo 文本 + Agnes LLM） |
| 文案匹配 | ✅ Chinese-CLIP 语义匹配 | ⚠️ 等分时间戳回填（M2.5） |
| 分镜工作台 | ✅ 表格核心编辑 + 视频联动 | ⚠️ 4 字段可编辑（M2.5 增强表格布局） |
| 配音系统 | ✅ 5 引擎（Edge/Index/CosyVoice/Volcano/OpenAI） | ⚠️ 单一 XiaomiMimo（M5 范围） |
| 导出 | ✅ 4 格式（剪映/PR/CapCut/SRT） | ✅ 剪映 JSON（M2 已有，M5 多格式） |
| 文案质检 | ✅ 六维评分 | ❌ 不做（M5+ 远期） |

### 2.2 两种核心创作模式

| 模式 | 适用 | 流程步骤 |
|------|------|----------|
| **A. 上帝视角（AI 帮我写）** | 没有文案，让 AI 自动看视频后生成解说词 | **8 步** |
| **B. 文案配视频（我有文案）** | 已有写好的文案，让 AI 自动匹配合适的视频画面 | **7 步** |

两种模式在 **4-7 步完全相同**（场景检测→分析→匹配→分镜），只在 **1-3 步** 区别。

### 3.2 八步详细

#### ① 选择视频 + 设置范围
- 拖拽/点击上传 mp4/mkv/avi/mov，最大 4GB
- 支持单视频 + 多集批量
- 设置视频范围（如 00:05:00 - 01:30:00）
- 显示已选视频：📁 电视剧第01集.mp4 · 02:35:18 · 3.2 GB

#### ② 配置解说参数
**风格选择**（15 种预设）：默认 / 电影解说 / 电视剧 / 综艺解说 / 动漫解说 / 短剧漫剪 / 高能合辑 / 口语化 / 萌讲八道 / 直播带货 / 新闻联播 / 吐槽式 / 深度解析 / 慢节奏 / 悬疑文案

**基础参数**：
- 解说时长：自定义 3 分钟（≈ 810 字 / 54 段）
- 解说语言：中文（13 种）
- 解说模型：默认（新手推荐）
- 分析模型：默认

**高级选项**：
- 解说辅助（选填）："重点讲主角成长线，弱化配角支线..."
- 智能穿插原片：✓ 开 / 密度 30%
- 文案质检：开

**测试连通**：验证 API Key 有效

**开始生成解说** 主按钮

#### ③ 场景检测（TransNetV2）
- 自动检测视频场景切换点
- GPU 加速 + 自动降级 CPU
- 进度显示：检测窗口 67/100 · 已用时间 2分13秒 · 发现 142 个场景
- 底部技术信息：GPU 加速模式 · TransNetV2 · NVIDIA GeForce RTX 4060
- 长视频自动裁剪分析范围（-ss/-t），只解码分析区间，提速 6-10x

#### ④ AI 视频内容分析
- 默认 Gemini 2.5 Flash 视觉模型
- 视频按时间线分块，并行 5 路处理
- 进度：分块 5/9 · 并行数 5 · 已识别 68 个场景
- 实时日志显示（monospace）：
  ```
  [场景 1] 00:00:00-00:01:23 · 城市外景，航拍镜头，高楼大厦...
  [场景 2] 00:01:23-00:02:45 · 主角在办公室打电话，表情焦虑...
  [场景 3] 00:02:45-00:03:18 · 会议室讨论，3人，激烈争论...
  [场景 4] 00:03:18-00:04:50 · 走廊追逐戏，动作紧凑...
  ```
- 智能分块→长视频按场景边界切（避免切割关键帧）
- 对冲请求加速尾块 + 熔断保护

#### ⑤ AI 生成解说文案
- 基于视频分析结果，**流式生成**解说文案
- **六段式故事结构**：开端→铺垫→冲突→高潮→反转→结局
- 进度：已生成 512 tokens · 流式输出中
- 文案示例：
  ```
  [00:01:23 - 00:02:45] 大家好，今天我们来聊这部电影...
  城市航拍缓缓拉开序幕，主角正坐在办公室里，眉头紧锁地看着手中的电话...
  [00:02:45 - 00:03:18] 突然，会议室的门被推开，三个人走了进来...
  ▌ 正在生成中...
  ```
- 12 种风格 Prompt 注入 + 时长动态换算字数
- 流式输出（streamGenerateContent）

#### ⑥ 文案-画面匹配
- 将 54 段解说词匹配到 142 个场景
- 进度：语义匹配 + 时间戳定位 + 覆盖审计
- 平均相似度 0.87
- 低相似度警告：场景 7 匹配度 0.62，建议人工调整

详见 §五

#### ⑧ 导出

- 📹 完整项目（配音 + 字幕 + 视频）
- 🎬 剪映草稿（导入剪映继续编辑）
- 🎞️ Premiere Pro XML

输出路径 + 视频质量（1080p/720p/480p）

### 4.2 关键区别
- 模式 A：AI 创作（场景→文案）
- 模式 B：AI 匹配（文案→场景）—— 文案来自用户，AI 的核心工作是**匹配**

### 4.3 模式 B 适用场景

## 七、配音系统

### 7.1 五大引擎

| 引擎 | 类型 | 速度 | 音色数 | 特点 |
|------|------|------|--------|------|
| **Edge TTS** | 云端 | 快 | 40+ 13种语言 | 微软免费、内置重试 |
| **IndexTTS** | 本地 GPU | 60-80% 加速 | 多 | 需 8G+ 显存 |
| **CosyVoice** | 阿里云 | 中 | 13 个龙系 | 高自然度 |
| **Volcano TTS** | 字节火山 | 中 | 多 | 声音复刻 ICL |
| **OpenAI TTS** | OpenAI | 中 | 6 | 商业 API |

### 7.2 单条配音流程
1. 用户选配音引擎 + 音色 + 语速
2. 点击「批量配音」/「单独配音」/「导入整段配音」
3. 后端调用 `tts_voice_gen` 任务
4. 实时进度：配音完成 48/54 · 待配 6 条
5. 生成的音频文件存到 `data_dir/voiceovers/{projectId}/{shot_id}.mp3`
6. 写入 `voiceovers` 表

## 八、导出系统

### 8.1 4 种导出格式

| 格式 | 文件 | 适用 |
|------|------|------|
| **完整项目** | mp4 + srt + audio 目录 | 直接发布的成片 |
| **剪映草稿** | draft_content.json + materials/ | 导入剪映 Pro 继续精剪 |
| **Premiere** | prproj + xml | 导入 Premiere Pro |
| **CapCut** | draft_content.json | 导入国际版剪映 |
| **SRT 字幕** | .srt | 仅字幕文件 |

### 8.2 剪映草稿结构
```json
{
  "app_version": "5.x",
  "fps": 30,
  "canvas": { "w": 1920, "h": 1080 },
  "duration": 754.0,
  "materials": {
    "videos": [...],
    "audios": [...],
    "texts": [...]
  },
  "tracks": [
    { "type": "video", "segments": [...] },
    { "type": "audio", "segments": [...] },
    { "type": "text", "segments": [...] }
  ]
}
```

### 8.3 视频质量选项
- 原画（1080p）
- 高清（720p）
- 标准（480p）

### 8.4 输出路径
- 默认：`D:\AIIA\output\<项目名>`（参考软件）
- VideosFlow：`data_dir/film_projects/<id>/output.mp4`（已有）

## 十二、关键决策点

### 12.1 VideosFlow 影片模块 MVP 范围（M2.5）
- ✅ ASR + LLM 真链路
- ✅ 6 段式 prompt
- ✅ 章节切分 + 时间戳回填
- ✅ 3 档降级
- ❌ 场景检测 / 视频分析（M6）
- ❌ 多配音引擎（M5）
- ❌ 多导出格式（M5）

### 12.2 关键 trade-off
| 选择 | 理由 |
|------|------|
| 6 段式 vs 4 段式 | 6 段式是参考软件标准，足够区分故事节奏 |
| 等分时间戳 vs LLM 自切 | ASR 无时间戳，LLM 自切更慢；等分快且确定 |
| 12 风格 vs 15 风格 | 少 3 个偏冷门（直播带货/深度解析/慢节奏），MVP 不做 |
| 3 分钟固定 vs 用户可调 | MVP 固定 3 分钟；后续在参数面板暴露 |

### 12.3 性能预算
- 3 分钟视频
- ASR（XiaomiMimo）：~30 秒
- LLM（Agnes 六段式）：~30 秒
- 切分 + 落库：<5 秒
- **总时长：~90 秒**

## 十三、风险与对策

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| XiaomiMimo ASR 仅返回整段文本（无时间戳） | 已确认 | 时间戳精度差 | **接受**：等分估算（接受精度限制） |
| Agnes LLM 返回非 JSON | 中 | 文案解析失败 | 降级：前 810 字 ASR 直接拼接 + 简单 4 段式切分 |
| 12 风格不够 | 低 | 用户反馈 | M5+ 扩 3 个 |
| 时间戳回填与音频时长不匹配 | 高 | 配音时序错位 | M5 字幕时轴对齐解决 |
| 6 段式对短视频（<1 分钟）不合适 | 中 | 文案过少 | M2.5 后期：检测视频时长自适应（<2 分钟合并为 3 段） |
| LLM 风格 prompt 注入不规范 | 中 | 文案质量差 | 用 settings.prompts.narration 模板（已存在），不要硬编码 |

## 十四、验收门槛（M2.5 完成时）

- [ ] **桌面端跑通**：上传 mp4 → 点"自动生成解说文案" → 4-6 段带时间戳的文案 + 分镜表 4-6 行
- [ ] **耗时**：3 分钟视频 < 90 秒
- [ ] **降级路径**：缺 Key / 网络断 → UI 显示"降级文案" + 任务栏警告
- [ ] **可重复**：已生成文案的工程再点按钮 → 重新生成
- [ ] **风格切换**：选不同风格（如"激情解说" vs "轻松搞笑"）→ 文案语气明显不同
- [ ] **时间戳对齐**：4-6 段时间戳与电影总时长（duration）等分，累加 ≈ duration
- [ ] **Mock 路径**：纯浏览器模式（无 Rust 后端）走 sim 占位，UI 不崩
